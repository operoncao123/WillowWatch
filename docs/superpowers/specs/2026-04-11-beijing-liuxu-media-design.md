# 北京柳絮多源传播版设计

## 背景

现有项目是单城市、纯静态、双卡片结构，只能展示北京市整体风险，不支持分区、不具备媒体传播感，也没有多源实时信号能力。新的目标是把它升级成一个面向移动端传播的北京柳絮专题页，覆盖北京 16 区，并由独立后端负责整合天气、生态暴露和社交热度三类信号。

## 产品目标

- 覆盖北京 16 区，支持全市概览与单区详情切换
- 首页具备小红书/媒体封面感，适合截图传播
- 风险分由天气、生态因子、社交热度共同决定
- 展示实时热议摘要与原帖卡片
- 前端为静态专题页，后端为独立 API 服务

## 非目标

- 不做用户上传图片与审核
- 不做街道级/小区级精准预报
- 不承诺社交抓取长期稳定可用
- 不把社交内容当作唯一风险来源

## 架构

系统拆为两层：

1. 静态前端：负责展示封面海报、16 区风险矩阵、单区详情、现场热议卡片。
2. Node 后端：负责天气采集、生态修正、社交采集、风险计算和 API 输出。

前端不直接抓小红书和微博内容，只请求后端聚合结果。这样能保持前端快、轻、可传播，同时让抓取与清洗逻辑留在服务端。

## 数据来源

### 天气

- Open-Meteo
- 按北京 16 区中心点分别请求天气

### 生态暴露

- 北京 16 区静态配置
- 区级生态权重
- 重点公园/滨河绿地/热点区域标签

### 社交信号

- 小红书
- 微博

社交采集属于真实抓取链路，但必须允许降级。若某个来源暂时失效，系统不能整体失效，而是退回天气 + 生态模型。

## 风险引擎

每个区的风险分由三部分组成：

- `weather_score`：气象风险基础分
- `ecology_multiplier`：生态暴露修正倍率
- `social_boost`：社交热度强修正

建议合成公式：

`final_score = weather_score * ecology_multiplier + social_boost`

然后限制在 `0-100`。

设计原则：

- 天气分仍是基础
- 生态因子体现区级长期暴露差异
- 社交信号是强修正，但必须限幅

## 社交信号策略

### 采集

- 关键词围绕“北京 + 柳絮/飞絮/杨絮 + 区名/公园/地标”
- 只看最近时间窗内容

### 清洗

- 去重
- 非北京内容过滤
- 广告/营销内容过滤
- 时间衰减
- 强度识别

### 区级归类

- 优先按区名
- 其次按地标/公园映射到区
- 无法归区的内容只记入全市热度

### 展示

前端展示两层：

- 热度摘要
- 原帖卡片

原帖卡片只展示必要字段：平台、时间、摘要、缩略图、跳转链接。

## 前端信息架构

### 1. 首屏海报

- 大标题偏媒体风
- 今日一句话结论
- 最高风险区
- 更新时间
- 传播型视觉

### 2. 16 区风险矩阵/榜单

- 强色块矩阵
- 一眼识别高风险区
- 点击切换当前区

### 3. 当前区详情

- 今日 / 明日风险
- 温度、湿度、风速、降水
- 气象驱动、生态修正、社交热度贡献
- 防护建议
- 重点留意区域

### 4. 现场热议

- 摘要
- 原帖卡片

### 5. 底部说明

- 模型说明
- 数据来源
- 免责说明

## 技术边界

- 前端继续使用原生 HTML/CSS/JS
- 后端采用 Node
- API 使用 JSON
- 允许增加最小必要依赖

## 风险

1. 社交抓取随平台变化而失效
2. 社交噪音带来误判
3. 生态因子只能做到区级合理估计
4. 海报感和信息密度存在天然冲突

## 验证

- 风险引擎单测
- 区级排序与权重单测
- 社交去重与归类单测
- API 响应结构测试
- 前端在移动端可读与可截图验证
